# RAG 问答系统项目分析

更新时间：2026-04-08

## 0. 当前仓库现状

- 当前 `RAGPro` 仓库基本为空，还没有现成代码、目录约定或本地项目 skills。
- 因此这份分析主要基于你提供的流程图，外加对关键技术选型的官方资料核验。

## 1. 项目目标

这张流程图对应的不是一个普通聊天机器人，而是一个“面向专业知识问答”的 RAG 系统。它试图把用户问题先做路由，再决定检索策略，最后结合知识库和大模型生成答案。

从产品能力上看，这个系统要解决的是：

1. 用户输入自然语言问题。
2. 系统判断这是不是知识库内的“专业咨询”。
3. 如果是专业咨询，进入检索增强回答链路。
4. 如果不是，走通用问答模型直接回答。
5. 专业咨询场景下，系统还要判断该用哪种检索策略。
6. 检索结果经过 rerank 和答案整理后，再输出最终答案。

## 2. 流程图模块拆解

### 2.1 在线问答链路

按流程图可拆成 6 个在线模块：

#### A. Query 接入层

职责：

- 接收用户 query
- 做输入清洗
- 记录 trace / session / query_id

建议补充：

- 敏感词与超长输入保护
- 问句标准化
- 日志埋点

#### B. 意图识别模块

流程图设计：

- 使用本地 `bert-base-chinese` 微调模型
- 把问题分成“专业咨询”和“非专业咨询”

这一步的本质是一个二分类路由器：

- 专业咨询 -> 走 RAG
- 非专业咨询 -> 走通用大模型

优点：

- 能避免所有问题都进入向量检索
- 可以降低非知识库问题对 RAG 召回链路的干扰

风险：

- 首版就单独训练分类模型，工程成本偏高
- 如果训练样本不足，容易把应检索的问题误判成闲聊

#### C. 检索策略选择模块

流程图设计：

- 专业咨询进入一个“策略选择模型（LLM + Prompt）”
- 然后再决定用哪种检索方式

这说明你的设计不是“单一召回”，而是“查询分析 + 动态检索策略”。

这部分很有价值，但也是首版最容易过度设计的地方。

#### D. 四类检索策略模块

流程图里至少表达了 4 类检索路径：

1. 直接检索
   适合事实明确、关键词清晰的问题。

2. 假设问题检索
   更接近 HyDE 思路，先生成假设答案，再拿假设答案去检索。

3. 子查询检索
   把复杂问题拆成多个子问题，再分别召回。

4. 回溯 / 问题细化检索
   对模糊问题先重写或澄清，再检索。

这 4 种方法分别对应：

- 直接事实问答
- 抽象问题补召回
- 多跳/组合问题
- 模糊表达优化

#### E. Milvus 检索与 rerank

流程图设计：

- 所有专业检索最终汇聚到 `Milvus`
- 然后经过 `rerank` 模型
- 再进入“问题整理模型（LLM + Prompt）”生成答案

这是很合理的：

- 检索负责“找材料”
- rerank 负责“精排材料”
- LLM 负责“组织答案”

#### F. 通识问答链路

流程图右侧保留了一个“通识问答模型（LLM + Prompt）”。

这意味着系统其实是“双通道”：

- 知识库增强问答
- 通用大模型问答

这个设计是对的，但要注意边界：

- 如果通识链路返回了未经知识库约束的答案，就不能和专业问答混在同一个“可信级别”上展示。

### 2.2 离线训练链路

流程图右上角是意图识别训练模块：

- 意图分类数据
- 预训练模型 `bert-base-chinese`
- 模型微调
- 模型评估
- 部署到线上做 query 路由

这条链路说明项目不是纯 prompt engineering，而是包含一个传统 NLP 分类子系统。

### 2.3 文档入库链路

流程图右下角可拆成：

1. FAQ 文档源
2. 多格式加载器
   - PDF
   - DOC
   - TXT
   - PPT
3. `document` 标准化
4. 父文块切分
5. 子块切分
6. 文档嵌入模型
7. 写入 Milvus

这说明你想做的是“父子块检索”而不是简单定长切块。

这个设计的价值在于：

- 子块适合召回
- 父块适合给 LLM 提供更完整上下文

这比只做固定长度 chunk 更适合 FAQ、政策文档、课程说明、制度文本等中文知识库。

## 3. 这个架构里“设计正确”的部分

以下部分我认为方向是对的：

### 3.1 把系统拆成“入库、检索、生成、分类”四层

这是一个成熟 RAG 项目应有的边界，不会把所有逻辑都堆进一个 prompt。

### 3.2 引入 rerank

如果没有 rerank，很多中文问答系统会出现：

- 召回看起来相关，但其实不够精确
- 多段内容都沾边，但最关键证据没排前面

rerank 往往比“继续加大模型”更能直接提升答案质量。

### 3.3 父子块切分思路合理

尤其是中文 FAQ、教务制度、培训资料、内部知识库，常常一段短句不足以回答问题，但整页又太长。父子块是个很实用的折中。

### 3.4 区分专业咨询与非专业咨询

这一步可以显著减少“不该查库的问题”进入检索链路，控制延迟和成本。

## 4. 这个架构里“首版偏重”的部分

如果直接照流程图完整实现，首版复杂度会很高。主要重在这里：

### 4.1 首版就训练本地 BERT 意图分类器

问题不在于它不能做，而在于它不是 MVP 的第一优先级。

首版更常见的做法是：

- 先用规则 + LLM 路由
- 收集真实 query
- 再训练分类模型替换 prompt 路由

原因：

- 真实线上问题分布和你预想的不一样
- 没有足够标注数据时，分类器效果常常不稳定
- 首版最关键的是把“可回答率”和“证据相关性”做出来

### 4.2 一次性实现 4 种检索策略

这是最大的复杂度来源。

虽然这 4 类策略都合理，但首版没必要一起上。因为你还不知道：

- 用户问题里复杂问题的占比多高
- HyDE 是否真的提升你的领域语料召回
- 子查询拆解会不会带来噪声
- 回溯重写是否会篡改原意

### 4.3 训练链路与在线链路耦合得太早

流程图把“意图识别训练模块”放进主架构是对的，但从工程节奏上，最好把它看成二期增强，而不是一期阻塞项。

## 5. 更适合落地的 MVP 方案

我建议把这张图收敛成一个三阶段方案。

### 阶段 1：先做最小闭环

目标：

- 文档可入库
- 问题可检索
- 检索结果可 rerank
- LLM 能基于证据回答

首版链路建议：

1. 多格式文档解析
2. 文档清洗与标准化
3. 父子块切分
4. 向量化并写入 Milvus
5. 用户 query 进入“轻量路由器”
6. 专业问题 -> 检索 + rerank + 生成
7. 非专业问题 -> 通用回答或安全兜底

阶段 1 不建议做：

- 独立训练 BERT 意图模型
- 4 套检索策略全部上线
- 复杂多跳编排

阶段 1 只保留两种检索策略即可：

1. 直接检索
2. 问题重写后检索

这两个就足够覆盖大部分 FAQ/知识库问答。

### 阶段 2：增强召回质量

在你有真实 query 日志后，再加：

1. 子查询拆解
2. HyDE / 假设答案检索
3. 元数据过滤
4. 会话上下文改写

### 阶段 3：把路由器从 prompt 升级为模型

这时再引入：

1. 意图分类数据集
2. `bert-base-chinese` 或更轻量中文分类模型微调
3. 离线评估
4. A/B 对比线上 prompt router

## 6. 推荐的系统模块设计

下面这套更适合你后续真正开始搭项目。

### 6.1 ingestion-service

职责：

- 读取 PDF / DOC / TXT / PPT
- 转成统一 `Document`
- 补充 metadata
- 产出 chunk / parent_chunk / child_chunk

关键字段建议：

- `doc_id`
- `source_type`
- `source_path`
- `title`
- `section`
- `parent_chunk_id`
- `chunk_id`
- `content`
- `keywords`
- `created_at`
- `version`

### 6.2 indexing-service

职责：

- 调 embedding 模型
- 生成向量
- 写入 Milvus
- 建立 chunk 与 parent_chunk 映射

### 6.3 retrieval-service

职责：

- 接收 query
- 执行 query rewrite
- 执行召回
- 合并候选集
- 调 reranker 精排

### 6.4 answer-service

职责：

- 组装最终 prompt
- 注入证据片段
- 生成最终答案
- 输出引用来源
- 在证据不足时拒答或降级

### 6.5 router-service

职责：

- 判断是否走专业 RAG
- 判断使用哪种检索策略

建议首版实现：

- 规则 + prompt router

而不是：

- 独立训练分类模型

### 6.6 evaluation-service

职责：

- 维护测试问题集
- 评估召回命中率
- 评估 rerank 后 top-k 质量
- 评估最终答案是否引用正确证据

这个模块在流程图里没有被明显画出来，但实际上非常关键。

## 7. 对关键技术选型的判断

以下判断结合了 2026-04-08 查阅的官方资料。

### 7.1 Milvus 适合做这个项目的向量库

依据官方文档，Milvus 支持：

- ANN 向量检索
- metadata filtering
- hybrid search
- BM25 / 稀疏检索
- reranking

这和你的流程图非常匹配，尤其适合后续从“单路向量召回”演进到“稠密 + 稀疏 + rerank”的检索链路。

如果你后面要做：

- 混合检索
- 元数据过滤
- 多召回合并
- rerank 精排

Milvus 是合适的。

### 7.2 BGE-M3 是一个合理但偏重的起点

根据 FlagEmbedding 官方资料，`BGE-M3` 的优势在于：

- 支持多语言
- 支持 dense + sparse 能力
- 官方明确推荐 “hybrid retrieval + re-ranking” 这条路线

这和你的图高度一致。

但工程上要注意：

- `BGE-M3` 比轻量 embedding 模型更重
- 如果你首版只是中文 FAQ 问答，算力预算有限时可能会觉得慢

因此建议是：

- 如果你目标是中文知识库 + 后续混合检索，保留 `BGE-M3`
- 如果你目标只是先跑通一个轻量原型，也可以先用更轻的中文 embedding 模型，后续再升级

### 7.3 reranker 应该保留，而且最好尽早接入

官方资料和经验都指向一个结论：

- embedding 检索负责高召回
- reranker 负责高精度

对中文知识问答来说，reranker 的收益通常非常直接。

建议默认链路：

1. topN 召回
2. rerank
3. 取 topK 作为生成上下文

### 7.4 `bert-base-chinese` 做意图分类可行，但不建议阻塞 MVP

Hugging Face 官方的 `Transformers` 文档明确支持用 BERT 跑文本分类微调，这条技术路线本身没有问题。

问题在于工程优先级：

- 可行，不代表首版必须先做
- 如果标注集不成熟，反而会让系统更脆弱

我的建议是：

- 一期：prompt router
- 二期：采集日志并标注
- 三期：BERT 分类器上线替换

## 8. 我对这张图的结构性优化建议

如果把这张图直接变成工程实现，我会做 5 个调整。

### 8.1 把“策略选择模型”改成“查询分析器”

不要让它只输出一个抽象策略名，而是输出结构化结果，例如：

- `intent`: domain / general
- `query_type`: factual / complex / ambiguous
- `needs_rewrite`: true / false
- `needs_decompose`: true / false
- `retrieval_mode`: direct / rewrite / decompose / hyde

这样更容易编排和调试。

### 8.2 通识问答增加安全边界

建议通识链路返回时显式标注：

- 该回答未基于知识库检索

否则用户会误以为所有答案都来自企业/学校/业务知识库。

### 8.3 增加引用与证据展示

流程图里有答案生成，但没有明确“引用输出”。

这是 RAG 系统里必须加的：

- 命中文档名
- 命中章节
- 命中片段
- 相似度 / rerank 分数

### 8.4 增加拒答与降级策略

当以下情况出现时，系统不能硬答：

- 召回证据不足
- rerank 分数过低
- 命中文档互相矛盾

应该有：

- 拒答
- 引导用户缩小问题范围
- 建议人工处理

### 8.5 增加离线评测集

至少要维护三类测试问题：

1. 直接事实问题
2. 复杂组合问题
3. 知识库外问题

否则你很难知道是：

- 检索不行
- rerank 不行
- prompt 不行
- 还是路由不行


## 12. 结论

这张流程图的总体方向是对的，而且明显不是“玩具级 RAG”：

- 有路由
- 有多策略检索
- 有 rerank
- 有父子块
- 有离线训练链路

但如果直接一步做到位，首版成本会偏大。

更稳妥的做法是：

1. 先做“文档入库 -> 检索 -> rerank -> 生成”的最小闭环。
2. 路由先用规则 + prompt，不要一开始就被 BERT 分类器卡住。
3. 检索策略先保留 `direct + rewrite` 两条主链。
4. 等有真实 query 数据后，再演进到子查询、HyDE、BERT 分类。

## 14. 新增设计图补充结论

本节基于目录 `基于RAG的问答系统图片流程/` 中新增的几张图：

- `项目大致流程.png`
- `融合后的系统.png`
- `基于模块化RAG的问答系统代码结构图.png`

这些图把原先的抽象流程进一步具体化了，尤其补充了：

- MySQL 精确匹配前置
- Milvus 混合检索 + rerank 细节
- 模块化代码结构
- 文档加载器与 OCR 处理链路

### 14.1 你现在的整体思路已经变成“双引擎问答”

从新增图来看，系统不再只是单纯 RAG，而是：

1. 先走 MySQL / FAQ 精确匹配
2. 命中阈值高于 0.85 时直接返回答案
3. 否则进入意图识别与 RAG 检索链路

这是一个非常实用的思路，适合教育 FAQ、制度问答、课程问答这类场景。

它的价值在于：

- 高频标准问答响应快
- 对格式化 FAQ 非常稳定
- 降低 LLM 与向量检索成本
- 可以把“明确标准答案”和“基于文档推理答案”分层处理

### 14.2 `项目大致流程.png` 对 MVP 非常有帮助

这张图表达的是一个更轻量的首版方案：

- 本地文档 -> 切分 -> embedding -> Milvus
- query -> embedding -> 检索 top-k -> query + doc 拼接 -> LLM 输出

这其实就是最适合先实现的第一阶段。

它说明一个重要事实：

- 你自己的设计里其实已经同时存在“完整版 EduRAG”与“最小闭环版 RAG”

如果我们要落地，应该优先以这张图作为一期，而不是直接以“融合后的系统图”作为一期。

### 14.3 `融合后的系统.png` 是“目标态”，不是“首版态”

这张图已经把多个子系统融合成完整架构，包含：

- MySQL FAQ 精确匹配模块
- BERT 意图识别分类器
- LLM 检索策略分类
- 4 类检索策略
- BGE-M3 稠密/稀疏向量生成
- Milvus 混合召回
- bge-reranker-large 精排
- LLM 最终答案生成

这个设计在方向上是通顺的，但工程上应该视为目标态系统。

更准确地说：

- `项目大致流程.png` 是一期
- `融合后的系统.png` 是二期到三期演进目标

### 14.4 MySQL 模块应保留，但建议重命名为“FAQ 精确匹配层”

从融合图看，这个模块用：

- 问答对数据
- BM25 或关键词匹配
- 阈值判断
- 命中后直接回答案

我建议工程上不要把它理解成“数据库方案 A”，而要理解成：

- FAQ 精确匹配层

因为关键不是存储在 MySQL，而是这个层的功能是：

- 对结构化问答做高置信命中
- 把固定标准答案从生成链路中分流出去

以后即使底层不是 MySQL，而是 PostgreSQL、SQLite 或 ElasticSearch，这一层的职责都不变。

### 14.5 `BGE-M3 + Milvus + reranker` 这条主检索链很合理

融合图里最成熟的一段其实是 Milvus 检索模块：

- 文档切片
- BGE-M3 生成稠密向量和稀疏向量
- 写入 Milvus
- 查询时分别召回 top-k
- 合并候选
- 用 `bge-reranker-large` 排序
- 取 top M 交给 LLM

这是一个标准而且先进的混合检索设计，和官方推荐路线是一致的。

这部分应该被保留为项目的核心竞争力。

### 14.7 但当前代码结构图也暴露出一个风险：核心编排过重
